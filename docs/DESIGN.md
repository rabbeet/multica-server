## Problem Statement

Сейчас всё работает локально на macOS: multica + 5 Pulse-clones + multica daemon. Задача: перенести на сервер. С телефона — брейнштормить идеи через web-UI multica (chat, удобно с мобильного экрана). С лэптопа — заходить через SSH в один из 5 Pulse-worktree'ев и реализовывать выбранный план. Плановой workflow: побрейнштормили 10 планов, утром сел за лэптоп, "возьми нерелизованный план", получил список, выбрал, пошёл делать. Все статусы корректно отображаются в multica.

## What Makes This Cool

- Phone = whiteboard. Сел на диван, /office-hours, готовый план в общую очередь.
- Laptop = workshop. SSH утром, /pickup-plan, на исполнение.
- Single source of truth — git plans-репо. Multica читает оттуда статусы, Pulse worktrees туда пишут изменения, история всегда есть.
- Two Claude accounts разделяют capabilities — brainstorm Claude умеет читать код и писать планы, не умеет деплоить или трогать прод.
- One bootstrap.sh = working stack from scratch за 20 мин. Никакого "я забыл как это настраивал".

## Constraints

- Существующий сервер (специфика TBD, скорее всего DigitalOcean Frankfurt по аналогии с Pulse production)
- Существующие 5 Pulse-clones на сервере (`~/coding/pulse{,2..5}`)
- Phone = iPhone (mobile-first брейншторм-интерфейс)
- Solo developer, no team
- Ключевая цель: минимум ручной настройки, максимум скорости пересборки с нуля

## Premises

(подтверждаются в plan-ceo-review)

1. 5 параллельных Pulse-worktree'ев нужны для изоляции контекста при параллельной работе
2. Multica web-UI — правильный брейншторм-интерфейс (vs custom mobile app, vs ChatGPT-style)
3. Два Claude-аккаунта (один brainstorm, один development) — правильное разделение
4. Git-репо — правильный source of truth для планов (vs multica DB, vs filesystem only)
5. Tailscale mesh — правильное сетевое решение (vs Cloudflare Access, vs reverse SSH)

## Approaches Considered

### Approach A: Custom multica-server bootstrap repo (RECOMMENDED)

Один git-репо с `bootstrap.sh`, docker-compose, всеми конфигами в версии. Multica только брейншторм. Pulse через SSH+tmux. `/srv/plans` — отдельный git-репо. Два новых skill'а: `/publish-plan` + `/pickup-plan`.

Effort: M (4-6 часов human, ~30-40 мин CC на boilerplate + skills)
Risk: Low (проверенные компоненты: Tailscale, Docker, git)
Pros: idempotent, восстановим за 20 мин, вся конфигурация в version control
Cons: ~10 файлов glue-кода (bootstrap.sh, scripts/, configs/) надо поддерживать
Reuses: gstack skills (существующие), Pulse worktrees, Mac mini-style сервер

### Approach B: Multica defaults + zero customization

Multica out-of-the-box. Никакого `/publish-plan` — просто вручную копировать файлы планов из multica workspace dir в Pulse worktrees при необходимости.

Effort: S (1-2 часа)
Risk: Med (ручной handoff = забытые планы)
Pros: минимум нового кода
Cons: phone → laptop handoff manual ("scp этот файл с сервера"), нет видимости статуса, нет git-audit trail планов
Reuses: multica defaults

### Approach C: Replace multica with custom mobile-PWA + git

Построить мини-PWA (Progressive Web App) которая проксирует Anthropic API + пишет в git. Никакого multica daemon, Postgres, Docker сложности.

Effort: L (1-2 дня)
Risk: High (custom код в critical path, бремя поддержки)
Pros: проще стек (1 сервис vs 3), полный контроль
Cons: переизобретает 90% ценности multica (workspace mgmt, multi-session, agent observability), opportunity cost на Pulse работе
Reuses: ничего — чистый лист

## Recommended Approach

A — Custom multica-server bootstrap repo.

Multica решает реальные проблемы (workspace management, observability агентских run'ов, multi-session) которые иначе пришлось бы переписывать. B наивно про handoff friction. C — это "build instead of compose", неверный выбор для solo-infra. A держит тебя в зоне силы (композитор существующих инструментов) и даёт полную воспроизводимость через bootstrap.sh.

## Architecture (locked в /office-hours)

```
┌─ BRAINSTORM ─────────────────────────────────────────────────┐
│ iPhone Safari → https://multica.<tailnet>.ts.net             │
│ → Multica web UI → daemon → Claude in workspace              │
│   (BRAINSTORM_API_KEY, gstack skills, /publish-plan)         │
│ → cd /srv/plans && git commit && git push                    │
└──────────────────────────────────────────────────────────────┘

┌─ DEV ────────────────────────────────────────────────────────┐
│ Mac iTerm (или iPhone Blink Shell аварийно)                  │
│ → ssh rabbeeet@server (Tailscale SSH)                        │
│ → tmux attach -t pulseN                                      │
│ → cd ~/coding/pulseN && claude (main API key)                │
│ → /pickup-plan → /executing-plans → /ship                    │
│ → При смене статуса — git commit/push в /srv/plans           │
└──────────────────────────────────────────────────────────────┘

Сервер:
  /srv/plans              — git-репо, single source of truth
  ~/coding/pulse{,2..5}   — Pulse worktrees (как сейчас)
  Tailscale SSH (mesh)    — единственный путь снаружи
  Multica + tinyproxy + Caddy в Docker
```

## Security Model (5 layers)

L1 **Filesystem**: brainstorm-Claude container — Pulse-репо `:ro`, `/srv/plans` `:rw`, `read_only: true` rootfs, `cap_drop: ALL`, UID 1001, никаких mount'ов `~/.ssh`, `~/.forge`, `~/.gh`, `**/.env*`.

L2 **Network egress**: tinyproxy с allowlist (`api.anthropic.com`, `github.com` для git push в plans, tailnet hosts). Container в Docker network без default gateway.

L3 **Claude permissions** (`settings.json`): deny ssh/scp/forge/curl/psql/etc, allow только Read/Edit в /srv/plans + /workdir/pulse.

L4 **DB access**: Никакого. Claude читает `docs/business/database-schema.md` (уже есть), не подключается к prod БД.

L5 **Git deploy keys**: SSH-ключ привязан только к plans-репо (Allow write access). Pulse-репо для этого ключа не существует.

## Network Access (Tailscale)

- Multica слушает только на tailnet IP (`100.x.y.z`), не на `0.0.0.0`
- TLS via MagicDNS (`multica.<tailnet>.ts.net`)
- Tailscale ACL: только `rabbeeet@github` имеет доступ к `tag:multica-host`
- Tailscale device approval включён (новые устройства требуют ручного approve)
- Key expiry 90 дней
- 3 ограничение снято — без жёсткого лимита (Tailscale free плёт до 100 устройств)

## Deployment (idempotent, scripted)

Repo `multica-server`:
```
multica-server/
├── bootstrap.sh                # Single entrypoint, idempotent
├── update.sh                   # Pull + restart + verify
├── teardown.sh                 # Чистая разборка
├── .env.example
├── docker-compose.yml          # multica + tinyproxy + caddy
├── config/{caddy,tinyproxy,tailscale,claude}/
├── scripts/01-99-*.sh          # Каждый шаг отдельным скриптом, idempotent
├── skills/{publish-plan,pickup-plan}/SKILL.md
└── systemd/                    # tmux-pulse@.service, pg-backup.timer
```

Disaster recovery times:
- VPS dies, fresh server: ~20 мин (auth-key + bootstrap.sh)
- Multica data lost: ~5 мин (pg_restore from daily backup)
- Plans repo lost: ~1 мин (git clone from GitHub)
- Tailscale compromised: ~10 мин (new tailnet + bootstrap.sh)

## Open Questions

1. **5 worktrees** — реально нужно столько, или 2-3 хватит? Если за последние 30 дней не было 3 параллельных задач — это над-проектирование.
2. **Cloud-hosted multica** опция существует? Если да — Approach A сильно упрощается.
3. **Postgres backup destination** — same server (риск: сервер умер = данные умерли) или off-server (S3-compatible, +$2/мес)?
4. **/publish-plan и /pickup-plan конкуренция** — что если два Pulse-worktree одновременно делают `git pull` и оба видят план как ready? Нужна оптимистическая блокировка через git.

## Success Criteria

- Свежий VPS → working stack за <20 мин (один `./bootstrap.sh` run)
- Phone → /office-hours → /publish-plan → /pickup-plan в Pulse → /executing-plans → /ship, end-to-end
- Plan исчезает из "ready" сразу как picked up (без double-pickup)
- Ноль секретов утекает в Claude API контекст brainstorm-аккаунта
- Ноль ручных шагов post-bootstrap кроме: Tailscale ACL JSON paste, GitHub deploy key registration (по одному разу)

## Reviewer Concerns (from CEO review + outside voice)

Surfaced in /plan-ceo-review, fixes specified for /plan-eng-review to pick up.

1. **Race condition spec (Section 1, finding #1)**: `/pickup-plan` должен использовать optimistic locking. Псевдокод:
   ```bash
   while true:
     git fetch origin main
     git reset --hard origin/main
     plan = pick_ready_plan_with_unblocked_deps()
     [ -z "$plan" ] && exit_no_ready_plans
     update_frontmatter status=in-progress assigned_to=$worktree
     git commit && git push --force-with-lease=main:HEAD~1
     [ $? -eq 0 ] && break    # claim succeeded
     # else: someone beat us, retry
   done
   ```
   Specification поднимается в /plan-eng-review.

2. **Telegram-on-shipped pipeline (cherry-pick #2)**: GH Action triggers on `push` events to `main`. Reads frontmatter из current и previous commits. Если `status: ready/in-progress → shipped` — curl Telegram Bot API. **Concrete workflow file** spec в /plan-eng-review.

3. **Read-only rootfs vs Claude Code (outside voice #3)**: Claude Code требует writable `~/.claude/` (state). План: `tmpfs` на `/tmp` и `/home/claude/.cache`, named volume `claude-brainstorm-config:/home/claude/.claude`. **Validation**: `99-verify.sh` запускает `docker exec multica-daemon claude --version` и проверяет exit code 0. Если упадёт — корректируем mounts.

4. **5 worktrees RAM budget (outside voice #4)**: User defended после challenge. **Validation в bootstrap**: после `07-pulse-worktrees.sh` запускаем `09-validate-resources.sh` который смотрит `free -m` и алертит если `< 6 GB available` после старта tmux-сессий. Если падает — заводим issue в TODOS «reduce to 3 worktrees».

5. **Skill failure modes (outside voice #5)**: `/pickup-plan` blocked_by graph validation. Cycle detection (`A blocked_by B blocked_by A`). Dangling check (`blocked_by: <id>` указывает на несуществующий плана). Spec в /plan-eng-review.

6. **Strategic baseline (outside voice #6)**: zero-infra (ChatGPT + Notes + git pull) не рассматривался в alternatives. **Reviewed and rejected**: 20% delta (agent observability, audit trail, automated handoff, Telegram closure, parallel worktrees) — критичны для use-case. Note for future retrospective: если через 6 месяцев `/publish-plan` использован <10 раз — переоценить.

7. **DR times aspirational (outside voice #7)**: 20/5/1/10 минут — это targets, не measured. **Validation**: после первого успешного deploy запустить timed teardown + bootstrap, записать реальное время в README.

## Eng Review Decisions (lock'd in /plan-eng-review)

### Architectural locks
- **Skill location**: bundled в `multica-server/skills/`. bootstrap.sh копирует на хост в `~/.claude/skills/` + bind в multica-контейнер на `/home/claude/.claude/skills/`. Один source of truth.
- **Pulse-worktree dev ports**: pulse=8000/5173, pulse2=8001/5174, pulse3=8002/5175, pulse4=8003/5176, pulse5=8004/5177. Auto-export через direnv `.envrc` per worktree.
- **Plans repo isolation**: 6 separate clones — `/srv/plans-multica/` для brainstorm-Claude (ro view + own clone), `/srv/plans-pulse{,2,3,4,5}/` для каждого Pulse worktree. Каждый делает fetch+rebase+commit+push в своём clone.
- **Git transport from container**: HTTPS + fine-grained PAT (`BRAINSTORM_PAT` в .env). PAT scope = только plans-репо, read+write. 90-day expiry, ротация. Bypass'ит SSH-vs-tinyproxy архитектурную проблему.
- **Plans dashboard hosting**: Cloudflare Pages + Cloudflare Access (free tier). OAuth через GitHub. URL `https://plans.<твой-домен>` или CF subdomain. Mobile UX лучше GH Pages.
- **Optimistic locking pseudocode (corrected)**:
   ```bash
   while [ $retry -lt 5 ]; do
     git -C $clone_dir fetch origin main
     git -C $clone_dir reset --hard origin/main
     EXPECT_SHA=$(git -C $clone_dir rev-parse origin/main)
     plan=$(yq_pick_ready_unblocked)
     if [ -z "$plan" ]; then exit_no_ready_plans; fi
     yq_update_frontmatter "$plan" status=in-progress assigned_to=$worktree
     git -C $clone_dir add "$plan" && git -C $clone_dir commit -m "claim: $plan"
     if git -C $clone_dir push --force-with-lease=main:$EXPECT_SHA origin main; then
       break
     fi
     ((retry++))
   done
   [ $retry -eq 5 ] && exit_retry_exhausted_message
   ```

### Bash hygiene fixes
- All scripts: `set -euo pipefail` + sourced `_lib.sh` for shared error handlers.
- All conditional checks: explicit `if ... then ... fi` instead of `&&`/`||` chains under `set -e`.
- All filenames with timestamps: `$(date +%Y%m%d-%H%M%S)` explicit format.

### Tooling deps in 01-host-deps.sh
- `yq` (mikefarah, NOT python3-yq) — для frontmatter parsing
- `tailscale` — Tailscale daemon
- `docker`, `docker-compose-plugin` — обязательно
- `tmux`, `mosh` — для SSH+persistent sessions
- `direnv` — для per-worktree port env vars
- `jq`, `curl` — uniformly available, sanity check at top of bootstrap.sh

### Idempotency hardening
- `03-tailscale-up.sh`: проверяет `tailscale status --json` jq на `BackendState == "Running"` ДО вызова `tailscale up` с auth-key. Auth-key используется только если daemon не authenticated.
- `04-claude-user.sh`: динамически находит свободный UID в диапазоне 1001-1099 (избегаем коллизий с systemd-coredump etc).
- Все scripts начинаются с idempotency guard: `if <уже сделано>; then exit 0; fi`.

### Validation hardening (99-verify.sh)
- Реальный write+read+delete test внутри multica-контейнера (НЕ просто `claude --version`):
  ```bash
  docker exec multica-daemon bash -c 'echo test > /home/claude/.claude/test_$$ && cat /home/claude/.claude/test_$$ && rm /home/claude/.claude/test_$$'
  ```
- Tailscale auth expiry watchdog: parsing `tailscale status --json` для KeyExpiry, если <14 days до expiry → Telegram alert.
- Plans repo deploy access: `git ls-remote https://...@github.com/rabbeet/plans.git` exit code 0.
- 5 Pulse worktrees: каждый имеет свой /srv/plans-pulseN clone, deploy key configured.
- Egress allowlist test: внутри container `curl -o /dev/null -w "%{http_code}" https://api.clickavia.com` → должен timeout/blocked.

### GitHub Action specifics

**`.github/workflows/notify-shipped.yml`**:
```yaml
name: Notify shipped plans
on:
  push:
    branches: [main]
concurrency:
  group: notify-shipped-${{ github.ref }}
  cancel-in-progress: false   # one notify per shipped, NEVER cancel
permissions:
  contents: read
jobs:
  notify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0   # full history for diff
      - run: |
          # diff between event.before and event.after handles squash-merge
          BEFORE="${{ github.event.before }}"
          AFTER="${{ github.event.after }}"
          # find plans whose status changed to shipped
          for plan in $(git diff --name-only $BEFORE..$AFTER -- '*.md'); do
            prev_status=$(git show $BEFORE:$plan 2>/dev/null | yq -f extract '.status' || echo "missing")
            curr_status=$(yq -f extract '.status' "$plan")
            if [ "$prev_status" != "shipped" ] && [ "$curr_status" = "shipped" ]; then
              title=$(yq -f extract '.title // .id' "$plan")
              pr=$(yq -f extract '.related_pr // ""' "$plan")
              curl -s "https://api.telegram.org/bot${{ secrets.TELEGRAM_BOT_TOKEN }}/sendMessage" \
                -d chat_id=${{ secrets.TELEGRAM_CHAT_ID }} \
                -d text="🚀 «$title» в проде → $pr" || echo "telegram failed (non-fatal)"
            fi
          done
```

**Cloudflare Pages deploy** заменяет GitHub Pages action — webhook trigger от plans-репо, CF Pages билдит Jekyll.

### Worktree parallelization

Implementation phases (parallel-able):

| Step | Modules touched | Depends on |
|---|---|---|
| Bootstrap scaffold (bootstrap.sh, scripts/0X-*) | scripts/, bootstrap.sh | — |
| Docker compose (multica + tinyproxy + caddy) | docker-compose.yml, config/{tinyproxy,caddy}/ | — |
| Skills bundle (publish/pickup/archive) | skills/ | bootstrap (для placement) |
| GitHub Actions (notify, build) | .github/workflows/ | — |
| Plan templates | plans-репо templates/ | — |
| Test plan (bats + 99-verify) | test/ | bootstrap, skills |

**Parallel lanes**:
- **Lane A**: bootstrap scaffold + scripts/0X (sequential within)
- **Lane B**: docker-compose + configs (independent of Lane A)
- **Lane C**: skills bundle (waits on Lane A only — needs paths from bootstrap)
- **Lane D**: GH Actions + plan templates (independent — другой репо)

Execution order: A + B + D в parallel (3 worktree'я). After A merged, C launches. After C merged → integration test.
